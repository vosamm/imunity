# Security And Privacy

## 1. 보안 우선순위

이 프로젝트는 암 진단과 관련된 민감한 맥락을 다룬다. 초기 MVP에서도 보안과 개인정보 최소 수집을 기본값으로 둔다.

우선순위:

1. API 키와 시크릿 보호
2. 사용자의 건강 관련 정보 최소 수집
3. 로그와 분석 도구에 민감정보가 남지 않게 하기
4. AI 프롬프트와 외부 API 전송 데이터 최소화
5. 공식 출처와 AI 요약 구분

## 2. API 키 규칙

- 로컬 키는 `.env`에만 둔다.
- `.env`는 커밋하지 않는다.
- `.env.example`에는 placeholder만 둔다.
- 키 값을 채팅, 문서, 이슈, 커밋 메시지, 로그, 스크린샷에 넣지 않는다.
- 배포 환경에서는 Vercel/Supabase/GitHub Secrets 같은 시크릿 저장소를 사용한다.

## 3. 로그 규칙

금지:

- 전체 요청 URL 출력
- query string 출력
- `serviceKey`, `Authorization`, API token 출력
- 사용자가 입력한 건강정보 원문 출력

허용:

- HTTP 상태 코드
- 내부 request id
- source type
- service id
- 실패 단계
- 재시도 횟수

현재 `national_welfare.py`, `local_welfare.py`는 HTTP 오류 시 전체 URL 대신 상태 코드와 reason만 출력하도록 변경되어 있다.

## 4. 개인정보 최소 수집

초기 MVP에서 사용자의 회원가입 없이 검색 가능하게 설계한다.

수집하지 않는 정보:

- 이름
- 주민등록번호
- 전화번호
- 상세 진단명
- 진단서 파일
- 병원명
- 치료 이력

필터에 필요한 경우에도 저장하지 않고 브라우저 상태 또는 익명 요청으로 처리한다.

## 5. 민감정보 주의

암 진단, 치료 상태, 건강 상태는 민감정보로 취급해야 한다. 실제 운영 전 개인정보보호 전문가 또는 법률 검토가 필요하다.

서비스 문구는 다음 원칙을 따른다.

- 복지 대상 여부를 확정하지 않는다.
- 의료 조언처럼 보이는 표현을 쓰지 않는다.
- 공식 기관 확인이 필요하다는 안내를 제공한다.
- AI 요약은 참고용임을 명확히 한다.

## 6. AI 사용 규칙

- AI에는 공공 복지 원문 데이터만 보내는 것을 기본으로 한다.
- 사용자가 입력한 건강정보를 AI로 보내야 하는 기능은 MVP 범위에서 제외한다.
- AI가 만든 분류/요약은 원문 링크와 함께 보여준다.
- AI 출력 실패 또는 JSON 파싱 실패는 전체 배치를 중단하지 않는다.

## 7. 배포 전 보안 체크리스트

- `.env`가 커밋되지 않았는지 확인
- secret scanning 결과 확인
- API 에러 로그에 키가 남지 않는지 확인
- 프론트엔드 번들에 서버 키가 포함되지 않는지 확인
- 외부 분석 도구를 붙이기 전 개인정보 영향 검토
- 쿠키/추적 도구 사용 시 고지와 동의 정책 검토
- 관리자 기능에는 인증과 권한 관리 적용

## 8. 사고 대응 기준

API 키 노출이 의심되면 즉시 다음 순서로 처리한다.

1. 노출된 키 폐기 또는 재발급
2. Git history, 로그, 배포 환경에서 노출 위치 확인
3. 새 키를 시크릿 저장소에만 반영
4. 재발 방지 문서와 코드 수정
