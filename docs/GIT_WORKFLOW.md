# Git And GitHub Workflow

이 문서는 팀의 기존 작업을 보호하면서 개인 개발을 계속하기 위한 Git/GitHub 작업 규칙이다.

## 1. 기본 원칙

- `main`은 팀 기준 브랜치다.
- `main`에는 직접 커밋하거나 직접 푸시하지 않는다.
- 개인 작업은 개인 브랜치에서만 진행한다.
- 원격에 올려도 되는 브랜치와 올리면 안 되는 브랜치를 작업 전에 확인한다.
- 커밋, 푸시, PR 생성은 사용자 또는 개발 리더의 명시 승인을 받은 범위에서만 한다.

## 2. 개인 작업 브랜치

Sewoong Oh의 개인 작업 브랜치는 아래 하나를 기본으로 쓴다.

```bash
sewoongoh/dev
```

처음 만들 때:

```bash
git checkout main
git pull
git checkout -b sewoongoh/dev
git push -u origin sewoongoh/dev
```

이후 작업할 때:

```bash
git checkout sewoongoh/dev
git pull
# 작업
git add ...
git commit -m "작업 내용을 설명하는 메시지"
git push
```

## 3. 브랜치 사용 규칙

| 브랜치 | 용도 | 직접 푸시 |
|---|---|---|
| `main` | 팀 기준 브랜치, 배포 또는 리뷰 완료 기준 | 금지 |
| `sewoongoh/dev` | Sewoong Oh 개인 개발/실험/문서 작업 | 허용 |
| `feature/*` | 팀에서 별도 기능 브랜치를 요구할 때 | 팀 규칙에 따름 |
| `fix/*` | 팀에서 별도 수정 브랜치를 요구할 때 | 팀 규칙에 따름 |

개발 리더가 별도 브랜치 전략을 정하면 그 규칙을 우선한다.

## 4. AI/Codex 작업 규칙

AI 또는 Codex는 다음 규칙을 반드시 따른다.

- 작업 시작 시 `git status -sb`와 현재 브랜치를 확인한다.
- 현재 브랜치가 `main`이면 코드 변경 전에 사용자에게 확인한다.
- 사용자가 개인 브랜치를 지정한 경우 그 브랜치에서만 작업한다.
- `main`으로 직접 `git push`하지 않는다.
- `main`을 되돌리는 작업도 사용자 또는 개발 리더의 명시 승인 없이 하지 않는다.
- 로컬 커밋은 사용자가 요청하거나 승인한 경우에만 만든다.
- 원격 푸시는 사용자가 브랜치명과 푸시 범위를 승인한 경우에만 한다.
- 작업 완료 보고에는 현재 브랜치, 커밋 해시, 푸시 여부를 포함한다.

## 5. PR 권장 흐름

개인 브랜치에서 작업한 변경은 다음 흐름으로 팀에 공유한다.

1. `sewoongoh/dev`에 커밋과 푸시
2. 변경 요약, 검증 결과, 보안/개인정보 영향 정리
3. 개발 리더에게 리뷰 요청
4. 필요하면 GitHub PR 생성
5. 승인 후 `main`에 merge

## 6. 사고 기록과 재발 방지

2026-07-09에 Codex가 명시 승인 없이 `main`에 커밋하고 푸시했다.

커밋:

```text
e818ada Validate one-item Mistral preprocessing
```

변경 내용:

- 중앙부처/지자체 Mistral 1건 검증 결과 문서화
- 공공 API 연결 오류 시 요청 URL이 traceback에 노출될 수 있는 문제 수정
- 관련 보안/데이터 파이프라인 문서 갱신

문제:

- 개발 리더 승인 없이 팀 기준 브랜치인 `main`에 직접 반영했다.
- 사용자의 개인 개발 흐름과 팀 리뷰 흐름을 구분하지 않았다.
- 커밋/푸시 전에 명시 확인을 받지 않았다.

재발 방지:

- Codex는 `main`에서 커밋/푸시하지 않는다.
- Codex는 기본 작업 브랜치를 `sewoongoh/dev`로 사용한다.
- `main` 변경이 필요하면 먼저 이유, 변경 범위, 되돌림 계획을 설명하고 승인을 받는다.
- 이미 `main`에 올라간 변경을 되돌릴 때도 임의로 처리하지 않고 개발 리더 결정에 따른다.
