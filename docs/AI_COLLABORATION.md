# AI Collaboration Guide

## 1. 목적

이 문서는 여러 사람이 vibe coding 방식으로 AI와 협업할 때, 프로젝트 맥락과 보안 기준을 잃지 않게 하기 위한 작업 규칙이다.

## 2. 작업 시작 전 읽을 문서

작업자는 아래 순서로 읽는다.

1. `README.md`
2. `docs/INDEX.md`
3. 작업 성격에 맞는 문서
   - 제품/화면: `docs/PRODUCT_SPEC.md`
   - 데이터 수집: `docs/DATA_PIPELINE.md`
   - 구조 변경: `docs/ARCHITECTURE.md`
   - 보안/개인정보: `docs/SECURITY_PRIVACY.md`
   - Git/GitHub 작업: `docs/GIT_WORKFLOW.md`
   - 일정/우선순위: `docs/ROADMAP.md`

## 3. AI에게 주는 기본 컨텍스트

새 작업을 시작할 때 AI에게 다음 내용을 먼저 알려준다.

```text
이 프로젝트는 암환자가 자신에게 맞는 중앙/지자체 공공 복지 정보를 쉬운 필터링으로 확인하는 웹서비스다.
보안과 개인정보 최소 수집이 최우선이다.
API 키와 사용자 민감정보는 코드, 로그, 문서, 채팅에 노출하면 안 된다.
저장소 문서가 현재 source of truth다.
작업 전 README.md와 docs/INDEX.md를 읽고, 관련 문서를 갱신하면서 구현해라.
```

## 4. 작업 규칙

- 변경 전 `git status -sb`를 확인한다.
- 현재 브랜치를 확인하고, `main`이면 코드 변경/커밋/푸시 전에 사용자에게 확인한다.
- `.env` 내용은 읽거나 출력하지 않는다.
- API 키가 필요하면 존재 여부만 확인한다.
- 외부 API 테스트는 1건부터 시작한다.
- AI API를 쓰는 작업은 비용이 발생할 수 있음을 명시한다.
- 기능을 바꾸면 관련 문서도 바꾼다.
- 보안 기준을 낮추는 변경은 하지 않는다.
- 기존 사용자 변경이 있으면 임의로 되돌리지 않는다.
- 개인 개발은 `sewoongoh/dev` 브랜치에서 진행한다.
- `main`에는 직접 커밋하거나 직접 푸시하지 않는다.
- 커밋/푸시/PR 생성은 사용자 또는 개발 리더의 명시 승인 후에만 한다.

## 5. 커밋 메시지 규칙

커밋 메시지는 “무엇을 왜 바꿨는지”가 드러나야 한다.

좋은 예:

```text
보안 로그 노출 방지 및 프로젝트 상태 문서화

HTTP 오류 시 API 키가 포함된 요청 URL이 출력될 수 있어 상태 코드 중심의 오류 처리로 변경했다.
```

피해야 할 예:

```text
update
fix
api key setup
```

## 6. PR 또는 작업 완료 보고 기준

완료 보고에는 다음을 포함한다.

- 변경한 파일
- 사용자에게 보이는 변화
- 보안/개인정보 영향
- 실행한 검증
- 남은 리스크 또는 다음 작업

## 7. 금지 사항

- 실제 API 키를 문서나 코드에 넣기
- `.env`를 커밋하기
- 전체 요청 URL을 로그에 남기기
- 암환자 대상 여부를 법적으로 확정하는 표현 쓰기
- AI 요약만 보여주고 공식 출처를 숨기기
- 대량 배치를 첫 실행으로 돌리기

## 8. 결정 기록

중요한 선택은 문서에 남긴다.

예:

- 왜 Next.js를 선택했는지
- 왜 DB를 Supabase로 선택했는지
- 왜 특정 필터를 MVP에 포함/제외했는지
- 왜 사용자 건강정보 저장을 하지 않는지
