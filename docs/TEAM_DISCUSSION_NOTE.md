## 1. 현재 프로젝트 상태

- 프로젝트 목적은 암환자와 보호자가 자신에게 맞는 중앙부처/지자체 공공 복지 정보를 쉽게 찾도록 돕는 것이다.
- 현재는 웹앱이 아니라 Python 기반 공공 복지 데이터 수집/전처리 프로토타입이다.
- 중앙부처 복지서비스 API 1건 조회를 확인했다.
- 지자체 복지서비스 API 1건 조회를 확인했다.
- Mistral 전처리 코드는 있으며, 중앙부처/지자체 각 1건 실행 검증을 완료했다.
- 아직 DB, 웹 프론트엔드, API 서버, 로그인, 배포는 없다.
- 보안과 개인정보 최소 수집이 최우선 원칙이다.

## 2. 이미 정리된 중요한 원칙

- `.env` 내용은 읽거나 출력하지 않는다.
- API 키는 코드, 문서, 로그, 채팅, 커밋 메시지에 남기지 않는다.
- 전체 요청 URL이나 query string을 로그에 남기지 않는다.
- 사용자의 진단명, 병력, 주민등록번호, 연락처 같은 민감정보는 MVP에서 수집하지 않는다.
- AI 요약과 분류는 참고용이며, 공공 원문 데이터를 대체하지 않는다.
- 복지 대상 여부를 확정 표현으로 안내하지 않는다.
- 개인 개발은 `sewoongoh/dev` 브랜치에서 진행한다.
- `main`에는 직접 커밋하거나 직접 푸시하지 않는다.

## 4. 데이터 문제에 대한 현재 판단

공공 복지 데이터는 API가 있어도 바로 사용자에게 보여주기 어렵다.

핵심 문제:

- 중앙부처와 지자체 데이터 필드가 다르다.
- 설명이 문장형이라 조건을 기계적으로 해석하기 어렵다.
- 암환자 관련 여부를 단순 키워드로 판단하면 오탐과 누락이 생긴다.
- 데이터가 많아지면 실시간 API 조회 방식은 느리고 불안정하다.

권장 구조:

```text
공공 API
  -> 원문 저장
  -> 공통 스키마 정규화
  -> 암환자 관련성/지원유형 메타데이터 생성
  -> 로컬 JSON 또는 DB 저장
  -> 검색/필터 UI
```

중요한 원칙:

- 원문은 반드시 보존한다.
- AI 결과는 원문을 덮어쓰지 않는다.
- 불확실한 데이터는 버리지 말고 `확인 필요` 또는 낮은 우선순위로 둔다.
- 필터는 처음부터 강한 제외보다 랭킹과 근거 표시 중심으로 설계한다.

## 5. 정확성 관련 판단

### 5.1 `영암` 같은 오탐 문제

단순히 `암`이라는 글자가 들어갔다고 암환자 관련 복지로 판단하면 안 된다.

예:

- `영암`은 지역명일 수 있으므로 암 질환과 관련 없는 경우가 많다.
- `암환자`, `악성신생물`, `항암`, `방사선치료`, `산정특례`, `중증질환`은 암 관련성이 높은 신호다.

따라서 암 관련성은 제목 하나가 아니라 지원대상, 선정기준, 지원내용을 함께 보고 판단해야 한다.

### 5.2 의미상 관련된 지원이 빠지는 문제

`경제지원`이라는 필터를 눌렀을 때 `가발 구매비 지원`이 빠지면 안 된다.

MVP에서는 embedding search까지 구현하기보다 다음 방식이 현실적이다.

- 규칙 기반 지원유형 태그를 먼저 만든다.
- `가발`, `의료비`, `치료비`, `간병`, `교통비`, `생계`, `긴급복지` 등을 경제/의료 부담 완화와 연결한다.
- 이후 시간이 생기면 AI 또는 embedding search로 의미 검색을 보강한다.

### 5.3 메타데이터 불일치로 유실되는 문제

AI나 규칙이 만든 메타데이터는 틀릴 수 있다.

따라서 메타데이터는 처음부터 강한 제외 조건으로 쓰면 위험하다.

강하게 제외해도 되는 조건:

- 지역이 명확히 다름
- 나이 제한이 명확히 맞지 않음
- 서비스 종료가 명확함

강하게 제외하면 위험한 조건:

- 암 관련성
- 지원유형
- 소득조건
- 대상자 조건
- AI가 만든 카테고리

## 6. 6시간 MVP 제안

팀 구성:

- 대학생 중초보 개발자 1명
- 초보 개발자 2명
- 모두 AI 사용 가능
- 작업 가능 시간 6시간

6시간 목표는 전체 서비스를 완성하는 것이 아니라, end-to-end MVP를 만드는 것이다.

목표:

```text
공공 API 수집
  -> 공통 스키마 정규화
  -> 암 관련성/지원유형 태깅
  -> 로컬 JSON 저장
  -> 웹 화면에서 필터/검색/상세 보기
```

이 정도면 스코프가 너무 작지 않으면서도 6시간 안에 완성 가능성이 있다.

## 7. 6시간 안에 하지 말아야 할 것

- 로그인
- 회원가입
- Supabase/Postgres 연결
- 배포 자동화
- 실시간 전체 공공 API 검색
- AI 챗봇
- 전체 공공데이터 수집
- 복지 대상 여부 확정 판정
- 사용자 민감정보 저장

이 항목들은 프로젝트 가치를 높여 보일 수는 있지만, 6시간 안에 넣으면 실패 확률이 높다.

## 8. 6시간 안에 해야 할 것

필수:

- 중앙부처/지자체 각 10~20건 수집
- 공통 스키마 JSON 생성
- 암 관련성 분류: `high`, `medium`, `low`, `exclude`
- 지원유형 태그: `의료비`, `생계`, `돌봄`, `심리`, `고용`, `가족`, `기타`
- `영암` 같은 문자열 오탐 방지
- `가발 구매비 지원` 같은 의미상 경제지원 항목이 누락되지 않도록 태그 보강
- 웹 화면 또는 정적 HTML 화면에서 검색/필터/상세 보기 제공
- 공식 문의처, 링크, 마지막 수집 시각 표시

가능하면 추가:

- Mistral 요약은 소량 샘플에만 적용
- AI 요약은 원문과 분리해서 참고용으로 표시
- 데이터 품질 한계와 향후 개선 방향 문서화

## 9. 추천 역할 분담

| 사람 | 역할 | 산출물 |
|---|---|---|
| 중초보 개발자 | 통합 리드/프론트 | 검색 화면, JSON 연결, 최종 통합 |
| 초보 개발자 1 | 데이터 파이프라인 | 수집, 정규화, JSON 저장 |
| 초보 개발자 2 | 분류/테스트/문서 | 암 관련성 규칙, 오탐/누락 테스트, 발표 메모 |

초보 개발자에게 큰 기능 전체를 맡기기보다, 파일 단위와 체크리스트 단위로 맡기는 것이 안전하다.

## 10. 추천 파일 구조

```text
welfare_schema.py
welfare_classifier.py
welfare_export.py
data/welfare_services.json
web/index.html
web/app.js
web/style.css
```

시간이 부족하면 Next.js보다 정적 HTML/JS가 더 안전하다.

팀이 웹앱 기술을 꼭 보여줘야 한다면 Next.js를 쓰되, DB와 서버 API는 생략하고 정적 JSON을 import/fetch하는 방식으로 제한하는 것이 좋다.

## 12. 성공 기준

아래가 되면 6시간 MVP로는 성공이다.

- 실제 공공 API 기반 데이터가 화면에 보인다.
- 중앙부처와 지자체 데이터가 같은 카드 구조로 보인다.
- 지역, 지원유형, 검색어, 암 관련성 필터가 작동한다.
- 각 결과에 암 관련성 판단 근거가 있다.
- 공식 문의처 또는 링크가 있다.
- 사용자의 민감정보를 저장하지 않는다.
- AI 요약과 공공 원문 근거가 구분된다.

## 13. 팀과 논의할 결정 사항

1. 6시간 MVP를 정적 HTML/JS로 만들지, Next.js로 만들지.
2. 데이터 저장을 JSON 파일로 할지, SQLite까지 넣을지.
3. 수집 건수를 중앙/지자체 각 10건으로 할지, 20건까지 늘릴지.
4. Mistral을 데모에 포함할지, 비용/시간 때문에 규칙 기반 분류만 쓸지.
5. `main`에 들어간 기존 커밋을 유지할지, 개발 리더 판단으로 되돌릴지.
6. 발표에서 “완성 서비스”가 아니라 “end-to-end MVP”로 설명할지.

## 14. 내 판단

현재 문서와 방향성은 나쁘지 않다. 오히려 보안, 개인정보 최소 수집, 원문 보존, 공통 스키마 같은 핵심 원칙은 잘 잡혀 있다.

다만 지금 가장 위험한 것은 스코프를 멋있게 보이려고 넓히는 것이다.

6시간 안에 가장 설득력 있는 결과물은 다음이다.

```text
중앙/지자체 공공 복지 데이터를 실제로 가져와서
암환자 관점으로 정규화/분류하고
사용자가 조건으로 검색해 볼 수 있는 화면까지 만든다.
```

이것은 작지 않다. 데이터 수집, 정규화, 분류, 검색, 보안 원칙이 모두 들어간 end-to-end 프로젝트다.
