# Data Pipeline

## 1. 목적

중앙부처/지자체 복지서비스 데이터를 안정적으로 수집하고, 암환자에게 관련 있는 정보를 검색 가능한 형태로 정리한다.

## 2. 입력 소스

| 소스 | 현재 파일 | 상태 |
|---|---|---|
| 중앙부처 복지서비스 API | `national_welfare.py` | 1건 조회 확인 |
| 지자체 복지서비스 API | `local_welfare.py` | 1건 조회 확인 |

환경변수:

- `DATA_GO_KR_KEY`
- `MISTRAL_API_KEY`

## 3. 처리 단계

```text
1. 목록 조회
2. 상세 조회
3. XML 파싱
4. 원문 필드 보존
5. 공통 스키마 정규화
6. 암환자 관련성 분류
7. 사용자 필터용 메타데이터 생성
8. 저장
9. 검색 API 제공
```

현재 구현은 1-3 단계 중심이다. Mistral 전처리 코드는 6-7 단계의 초기 형태이며,
중앙부처/지자체 각 1건 실행 검증을 완료했다.

## 4. 공통 스키마 초안

| 필드 | 설명 |
|---|---|
| `source_type` | `national` 또는 `local` |
| `source_service_id` | 공공 API의 원본 서비스 ID |
| `title` | 서비스명 |
| `summary` | 서비스 요약 |
| `target` | 지원대상 |
| `criteria` | 선정기준 |
| `benefit` | 지원내용 |
| `application_method` | 신청방법 |
| `contact` | 문의처 |
| `links` | 공식 링크 목록 |
| `ministry` | 중앙부처/소관기관 |
| `region_sido` | 시도 |
| `region_sigungu` | 시군구 |
| `support_categories` | 치료비, 생계, 돌봄 등 분류 |
| `cancer_relevance` | `high`, `medium`, `low`, `exclude` |
| `cancer_relevance_reason` | 암환자 관련성 판단 근거 |
| `raw_payload` | 원문 응답 또는 원문 파싱 결과 |
| `fetched_at` | 수집 시각 |
| `processed_at` | 전처리 시각 |

## 5. 암환자 관련성 분류 기준 초안

`high`:

- 암, 악성신생물, 중증질환, 희귀난치, 산정특례, 항암, 방사선치료 등 직접 표현이 있다.
- 의료비, 치료비, 재난적 의료비 등 암 치료 비용과 직접 연결될 가능성이 높다.

`medium`:

- 질병, 장애, 입원, 돌봄, 생계, 저소득 의료 지원처럼 암환자에게 적용 가능성이 있으나 직접 표현은 없다.

`low`:

- 일반 복지이지만 암환자도 조건에 따라 볼 수 있는 제도다.

`exclude`:

- **암/질병/의료 맥락이 전혀 없는** 제도만 exclude한다 (예: 창업 융자, 영농 정착, 검정고시 축하금).
- ⚠️ 인구집단(아동·청소년·임산부 등)만을 근거로 exclude하지 않는다.
  소아암 환자·환아 가족 지원이 배제되는 것을 막기 위한 결정이다 (`docs/adr/0001-mvp-storage-stack-classification.md`).
  골든 케이스는 `acceptance_tests/test_phase2_classify.py`에 고정되어 있다.

분류 결과는 사용자에게 확정 판정으로 보여주지 않는다. 검색 랭킹과 필터링에만 사용한다.

## 6. 필터링 규칙

- 지역 조건이 없는 중앙부처 제도는 모든 지역에서 노출 가능하다.
- 조건이 명시되지 않은 필드는 “제한 없음”이 아니라 “원문에서 확인되지 않음”으로 저장한다.
- 사용자 필터에서 미입력 값은 결과 제외 조건으로 쓰지 않는다.
- 소득/나이/성별 조건은 파싱이 불확실하면 제외하지 말고 “확인 필요”로 표시한다.

## 7. AI 전처리 규칙

- AI는 원문을 요약하고 태그를 제안한다.
- AI 출력은 원문을 대체하지 않는다.
- JSON 파싱 실패는 해당 레코드만 실패 처리한다.
- 프롬프트에는 API 키나 불필요한 사용자 입력을 포함하지 않는다.
- 비용 관리를 위해 전체 배치 전 1건, 10건, 100건 순서로 검증한다.

|---|---|---|---|
| 중앙부처 | `WLF00000022` | 산재근로자 사회심리재활지원 | 성공, JSON 파싱 성공 |
| 지자체 | `WLF00006549` | 학교 밖 청소년 검정고시 합격 축하금 지원 | 성공, JSON 파싱 성공 |

확인된 점:

- 공공 API 목록 조회, 상세 조회, Mistral 호출, JSON 파싱까지 1건 기준으로 동작한다.
- 중앙부처와 지자체 프롬프트 모두 유효한 JSON을 반환했다.
- 현재 AI 출력은 공통 스키마와 직접 일치하지 않는다. 예를 들어 일부 값은 문자열이고 일부 값은 중첩 객체다.
- 다음 단계에서는 AI 출력 형태에 의존하지 않고, 원문 파싱 결과를 기준으로 공통 스키마 정규화 계층을 별도로 구현해야 한다.
- 암환자 관련성 분류는 아직 검증하지 않았다.

## 9. 완료 기준

수집 파이프라인은 다음 조건을 만족해야 운영 가능하다.

- 중앙/지자체 각 10건 이상 수집 및 상세 조회 성공
- 실패 레코드가 있어도 배치가 계속 진행됨
- 원문과 정규화 결과가 함께 저장됨
- 재실행 시 중복 데이터가 통제됨
- 로그에 API 키가 남지 않음
