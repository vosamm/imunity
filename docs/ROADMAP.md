# Roadmap

## Phase 0. 현재 상태 정리

상태: 진행 중

완료:

- 저장소 clone
- `.env` 로컬 설정
- 중앙부처 API 1건 조회 확인
- 지자체 API 1건 조회 확인
- API 오류 로그에 키가 노출될 수 있는 문제 수정
- 기본 문서와 보안 원칙 정리
- Mistral 전처리 1건 검증 (중앙/지자체 각 1건 성공)
- 수집/전처리 버그 수정: 페이지네이션 중복, 실패 레코드 스킵, 에러 XML 감지, 재시도 정책 통일
- 전처리 결과 `data/` JSON 저장 추가 (정식 저장 구조 전 임시 보존용)
- 수집/전처리 unittest 추가

남은 작업:

- 수집 결과 저장 방식 결정 (JSON vs SQLite)

## Phase 1. 데이터 파이프라인 안정화

목표: 중앙/지자체 데이터를 반복 수집하고 저장할 수 있게 한다.

작업:

- 공통 스키마 구현
- 중앙/지자체 수집 결과를 같은 구조로 정규화
- 로컬 JSON 또는 SQLite 저장 추가
- 중복 방지 키 정의
- 실패 레코드 로깅
- 10건, 100건 순차 테스트

완료 기준:

- 중앙/지자체 각 10건 이상 수집 성공
- 실패해도 전체 배치가 중단되지 않음
- 로그에 API 키가 없음

## Phase 2. 암환자 관련성 분류

목표: 암환자에게 관련 있는 복지 정보를 우선 노출할 수 있게 한다.

작업:

- 키워드 기반 1차 분류
- Mistral 기반 요약/분류 검증
- `high`, `medium`, `low`, `exclude` 기준 튜닝
- 분류 근거 저장

완료 기준:

- 각 결과에 관련성 등급과 근거가 있음
- AI 실패 시 키워드 기반 fallback 가능
- 사람이 샘플 30건을 검토할 수 있음

## Phase 3. 웹 MVP

목표: 사용자가 조건을 입력하고 복지 정보를 볼 수 있는 웹 화면을 만든다.

작업:

- Next.js 앱 생성
- 검색/필터 UI
- 결과 목록
- 상세 화면
- API route 또는 임시 JSON 데이터 연결
- 반응형 모바일 UI

완료 기준:

- 사용자가 지역/나이/소득/키워드로 결과를 좁힐 수 있음
- 각 결과에 공식 문의처 또는 링크가 있음
- 최신 수집 시각이 보임

## Phase 4. DB와 배포

목표: 로컬 프로토타입을 배포 가능한 서비스로 만든다.

작업:

- Postgres/Supabase 스키마 설계
- 수집 배치와 DB 연결
- Vercel 배포 설정
- 환경변수/시크릿 설정
- 기본 모니터링

완료 기준:

- 배포 URL에서 검색 가능
- 서버 키가 프론트엔드에 노출되지 않음
- 배치 재실행 가능

## Phase 5. 운영 품질

목표: 실제 사용자에게 보여줄 수 있는 안정성과 신뢰성을 확보한다.

작업:

- 관리자 수집 상태 화면
- 데이터 검수 워크플로우
- 접근성 점검
- 개인정보/법률 검토
- 사용자 피드백 반영

완료 기준:

- 데이터 오류 신고/수정 흐름이 있음
- 개인정보 처리 방침과 이용 안내 초안이 있음
- 민감정보 저장 여부가 명확히 통제됨
